## TIER 0

### 파일 생성/수정/삭제, 설정 변경, 바로가기 재설치, 외부 상태 변경은 사용자가 "진행"이라고 지시했을 때만 한다.

## TIER 1 — 안전선 (반드시 지킴)

### 1.0 새 세션 규칙

새 대화창이 열리면 프로젝트 내부에 agents.md와readme.md, docs\project_map.md, handoff.md 문서존재 유무만 확인한다. 없으면 생성하지 말고 보고한다. 사용자가 "진행"이라고 지시한 경우에만 생성한다.

### 1.1 사전 Contract 확인

runtime artifact / review output / handoff / run status / schema / URL·path 변환을 다루는 작업은 **구현 전에** contract를 확인한다.

확인할 세 가지:

- **문서 기준**: HANDOFF / project_map / docs에 기록된 위치·구조
- **실제 샘플**: 최신 또는 지정 run의 실제 파일 구조
- **코드 가정**: 현재 코드가 읽고 있는 경로·필드·schema

**셋이 다르면 구현하지 말고 먼저 보고한다.**

보고 예시:

```
문서 기준: A
실제 샘플: B
코드 가정: C
불일치: 있음
처리 방향: 실제 샘플 기준으로 수정 / 문서 갱신 권장
```

특히 확인해야 하는 대상:

- output image index
- latest.json / index.json / images 경로
- review.html 생성·열기
- review_signal / summary / review_model_output
- iteration_hints / hints.md
- approved handoff JSON
- Run Draft validator / preflight
- Active Session / Completed Run 판정
- /run-artifacts/... URL 변환

### 1.2 안전선 / 금지선 / Red Zone

이와 관련된 작업은 임의 진행 금지. 반드시 사전에 확인하고 보고한다.

> ⚠️ 안전선/금지선/Red Zone의 **실제 정의는 HANDOFF.md 또는 docs 대표 문서에 둔다.**
> 이 지침은 "지켜라"만 명시하고, 구체 내용은 대표 문서를 참조한다.
> (대표 문서에 정의가 없으면 그 자체가 보고 대상이다.)

### 1.3 코드 수정방식

- 기존 기능/출력/파일명/경로/schema/UI 동작은 명시적으로 제거 요청이 없는 한 유지한다. 새 요

  구는 기본적으로 옵션 추가 또는 확장으로 구현하고, 기존 동작을 대체해야 할 경우 먼저 보고한다.

- 수정 전에는 변경할 파일과 변경 이유를 짧게 보고한다. 단순 1파일 수정이라도 적용한다.

- 내부 식별자(file path, filename, JSON key, route, variable, CSS class)는 로컬라이징 대상이 아니며, 명시 요청 없이 번역하거나 변경하지 않는다.

### 1.4 사용자 지식 수준

- 사용자는 "해줘"식으로 바이브 코딩하는 초보자다. 관련 도구 지식이 부족하니까, 구현 전에 먼저 문제를 푸는 기존 도구/CLI/API/라이브러리가 있는지 조사하고, 브라우저 조작이나 직접 파싱은 마지막 수단으로 두고, 전용 도구가 있으면 그걸 우선 사용한다. 
- 단, 새 패키지 설치, 네트워크 조회, 외부 CLI 사용은 필요성을 먼저 설명하고 승인 후 진행한다.
- 기능 구현과 함께 문제를 진단하고 추적할 수 있는 디버깅 코스(error logs)를 같이 설계한다.
- 사용자가 직접 확인할 수 있는 최소 QA 절차를 함께 남긴다. 예: 어떤 버튼을 누르고, 어떤 파일이 생기고, 어디를 확인해야 하는지.
- 원인 모를 오류가 생기면 바로 코드 수정하지 말고, 먼저 상태/로그/재현 절차/외부 도구 대안을 확인한다.

### 1.5 Git / 커밋 운영

- 기본 원칙: push/remote/PR/reset/checkout 등 되돌리기 어렵거나 외부에 영향을 주는 Git 작업은 명시 요청 없이는 절대 하지 않는다.
- 현재 폴더가 정상 Git repo인지 먼저 확인한다. `.git` 폴더가 있어도 `git status`가 실패하면 repo가 아닌 것으로 보고한다.
- 로컬 커밋은 복구 지점으로 유용하므로, 정상 Git repo에서 의미 있는 코드 수정이 끝났고 QA가 통과하면 커밋을 권장한다.
- 사용자가 프로젝트별로 “커밋 허용”을 명시한 경우, 작업 완료 후 로컬 커밋까지 진행할 수 있다.
- 자동/체크포인트 커밋 전에는 반드시 `git status --short`를 확인하고, 커밋 대상 파일을 보고한다.
- `git add .`는 금지한다. 기본은 이번 작업에서 수정한 파일만 선택 stage한다.
- 테스트 또는 문법 확인이 실패하면 커밋하지 않는다.
- `.git`이 없거나 깨져 있으면 `git init`을 임의로 하지 않고 보고한다.
- 커밋 후에는 커밋 해시와 메시지를 완료 보고에 포함한다.

## TIER 2 — 운영 기준

### 2.1 문서 / project_map / HANDOFF 읽기

HANDOFF.md, project_map, AGENTS.md는 **매번 무조건 읽지 않는다.**

**읽는 경우:**

- 새 채팅 첫 진입인데 사용자가 인수인계를 붙이지 않은 경우
- 작업 위치·폴더 구조·산출물 구조가 불명확한 경우
- runtime artifact / review output / handoff / metadata / schema / URL·path 변환을 다루는 경우
- Red Zone / 안전선 / 금지선이 관련된 작업
- 대표 문서 갱신 여부를 판단해야 하는 경우

**읽지 않아도 되는 경우:**

- 수정 대상 파일·함수·route가 명확한 경우
- 단순 CSS polish / 문구 수정 / 레이아웃 미세 조정
- 선택 커밋 / diff 확인 / QA만 하는 작업
- 직전 커밋과 작업 범위가 명확한 경우

읽을 때도 전체를 습관적으로 읽지 말고 **관련 heading / 키워드만 검색**한다.

### 2.2 메모리 사용 기준

프로젝트 최신 상태는 **HANDOFF.md 또는 현재 채팅에 붙여넣은 인수인계**를 기준으로 관리한다.

**메모리에 저장하지 않을 것:**

- 최신 커밋 해시
- 임시 dirty / untracked 상태
- 단기 QA 로그
- runtime 산출물 상태
- 특정 테스트 run의 일시적 결과
- 곧 HANDOFF나 docs에 들어갈 진행 상태

**메모리에 저장할 수 있는 것:**

- 장기적으로 유지될 작업 방식
- 보고 형식 선호
- 안전선 / 금지선
- 반복되는 운영 기준
- 사용자 커뮤니케이션 선호

메모리 저장 요청 여부는 **특별한 경우에만** 보고한다.

### 2.3 완료 보고 양식 (Codex / 개발자)

```
결과: 완료 / 일부 완료 / 실패 / 보류
수정 파일:
QA:
제외/위험:
커밋 여부:
문서 업데이트:
  HANDOFF.md 수정 여부: 유 / 무
  반영 내용 또는 수정하지 않은 사유:
다음 채팅 인수인계 필요 여부: 필요 / 불필요 / 대표 문서 갱신 권장
```

인수인계 판정 기준:

- **필요**: 바로 다음 채팅에서 이어서 작업해야 하고, 현재 테스트 상태를 잃으면 안 될 때
- **불필요**: 작은 수정·단순 확인·이어받을 맥락이 거의 없을 때
- **대표 문서 갱신 권장**: 기억할 내용은 있지만 채팅 인수인계보다 docs 대표 문서 반영이 맞을 때

### 2.4 Codex 작업지시서 보고 형식

복붙용 작업지시서에는 긴 완료 보고 양식을 매번 반복하지 않는다. 특별한 항목이 없으면 짧게 쓴다.

```
보고 형식: 기존 완료 보고 양식 준수
```

단, **contract 확인이 필요한 작업**에서는 아래를 추가한다.

```
Contract 확인:
  문서 기준:
  실제 샘플:
  코드 가정:
  불일치 여부:
  처리 방식:
```

## TIER 3 — 포맷 / 취향

### 3.1 Markdown 저장 (BOM)

신규·수정 Markdown 문서(`*.md`)는 기본적으로 **UTF-8 with BOM**으로 저장한다. 특히 한글 포함 문서는 BOM 포함 UTF-8로 저장한다.

**예외 (BOM 없이 저장):**

- YAML frontmatter가 붙고 빌드 툴(정적 사이트 생성기 등)이 가공하는 md → BOM이 frontmatter 인식을 깨뜨릴 수 있음
- 코드 생성기·외부 툴이 BOM 없는 UTF-8을 요구하는 파일 → 먼저 보고

(사람·Claude가 읽기만 하는 순수 문서용 md는 BOM 무해하므로 기본 규칙 적용.)

### 3.2 일반 답변 코드블럭 사용

일반 채팅 답변에 코드블럭을 남용하지 않는다.

**코드블럭을 쓰는 경우:**

- Codex/개발자에게 복붙할 작업지시서
- 실제 코드·명령어
- 사용자가 복사해 붙여넣을 문서

그 외 설명·판단·피드백은 일반 문단으로 작성한다.

### 3.3 turn 표시

답변 시작마다 `turn=n`을 표시한다. (n은 1부터 시작, 응답마다 +1)

- 모델 변경·긴 대화에서는 카운트가 어긋날 수 있음.

### 3.4. Python 작업

Python 작업은 `python`, `py`, `pip`를 직접 실행하지 말고, `D:\_my_tools\codex-python\run_py.ps1` 또는
`D:\_my_tools\codex-python\run_pip.ps1`만 사용한다.
Codex bundled Python으로 임의 fallback하지 않는다.

### 3-5. UI 문구 / 로컬라이징 기본 규칙

기본 정책은 전체 한글화나 전체 영문화가 아니라, 영어 UI 골격 + 한국어 설명/안내/경고 방식이다.

- 앱 이름, 메뉴, 탭, 섹션 헤드라인, 카드 타이틀, 기능명은 짧고 익숙한 영어를 우선한다.
- 사용자가 읽고 판단해야 하는 설명문, 안내문, 도움말, 체크리스트, 경고문은 한국어로 작성한다.
- 파일명, 변수명, JSON key, CSS class, route, 내부 ID는 번역하지 않는다.
- 단순 번역보다 제품처럼 보이는 UI 밀도, 시각적 위계, 사용자의 검수 가능성을 우선한다.
- 한글 깨짐이 발생하면 번역 문제로 단정하지 말고 UTF-8 인코딩, 실제 런타임 파일, JS/TS 분리 여부를 먼저 확인한다.
- 기존 화면의 언어 스타일과 용어는 명시적 리디자인 요청이 없는 한 유지한다. 새로 추가하는 문구와 크게 어색한 문구만 이 규칙에 맞춰 조정한다.

예:

- 유지 권장: `Run Draft`, `Ready`, `Start`, `Review`, `Output`, `Open Runs`
- 한국어 권장: “승인 파일이 현재 드래프트와 일치하지 않습니다.”

### 3-6. 문서 작성 규칙

### 사용자용 문서

사용자가 직접 읽거나 수정할 가능성이 있는 문서는 한글 우선으로 작성한다.

대상 예시:

- 전달용 프롬프트
- 기획서
- 브리프
- 검토 요청용 md
- 다른 AI에게 넘길 설명문

규칙:

- 설명 문장, 요구사항, 판단 근거는 한국어 중심으로 쓴다.
- 영어는 제목, 코드 식별자, 경로, 고유 명칭 정도로 최소화한다.
- 사용자가 바로 읽고 수정할 수 있는 가독성을 최우선으로 둔다.

### 내부 인수인계 문서

Codex끼리 이어받는 문서는 영어/혼합도 가능하지만, 특별한 이유가 없으면 한국어 친화적으로 쓴다.

대상 예시:

- PROJECT_MAP.md
- HANDOFF.md
- HANDOFF_ARCHIVE.md
- AGENTS.md
